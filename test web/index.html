Efficient Single-GPU Performance Modeling for LLM Training Time Prediction Using RTX 4090 Abstract: Training Large Language Models (LLMs) has emerged as one of the most computationally intensive and resource-demanding processes in modern artificial intelligence. With models scaling to billions of parameters, accurately estimating training time is crucial for efficient resource allocation, cost optimization, and experimental planning. Existing approaches for predicting training time rely either on full-system simulation, which is computationally expensive, or partial execution, which incurs significant cost and time overhead. In this paper, we propose a comprehensive operator-level performance modeling framework for predicting end-to-end LLM training time on a single GPU, specifically the NVIDIA RTX 4090. Our approach decomposes the LLM computation graph into atomic operators, performs extensive micro-benchmarking across diverse configurations, and employs machine learning-based regression models to estimate operator runtime. By leveraging the repetitive structure of transformer architectures and simplifying distributed communication overhead into a single-GPU paradigm, we significantly reduce modeling complexity while maintaining high accuracy. We validate our framework using Qwen3.5-0.8B and demonstrate realistic use-case scenarios with varying token scales (10B–100B tokens). Our approach provides a scalable, efficient, and practical alternative to traditional methods, enabling accurate prediction without executing expensive training runs. Introduction The rapid evolution of Large Language Models (LLMs) has fundamentally transformed artificial intelligence. However, training these models is extremely expensive in terms of both time and computational resources. Predicting training time beforehand is therefore critical for planning, budgeting, and efficient system utilization. Despite its importance, training time prediction remains challenging due to heterogeneous workloads, hardware dependency, and complex execution pipelines. Existing solutions such as simulation-based and sampling-based approaches are either too slow or too costly. This work proposes a novel operator-level modeling framework that eliminates the need for simulation or execution by learning runtime behavior directly from micro-benchmarks. Purpose: Motivate need for prediction by showing how expensive real training is. Recent empirical training runs on large-scale HPC systems further highlight the importance of accurate training-time prediction. For example, pretraining a 500M-parameter transformer model on 8× NVIDIA A100-SXM4-80GB GPUs requires over 136 days to process 100 billion tokens, even with optimized data-parallel execution. Such prolonged runtimes make trial-and-error configuration tuning infeasible, reinforcing the need for accurate, execution-free performance modeling. II. This is difficult due to heterogeneous operations, nonlinear hardware behavior, and limitations of existing methods. We reformulate the problem as a data-driven operator-level prediction task. III. Proposed Approach: We model training as a composition of atomic operators and use regression models to predict their runtime. Transformer repetition allows scalability. IV. Methodology: We formalize our approach as a structured pipeline consisting of six key stages: Step 1: Choose LLM Model We begin by selecting the target LLM architecture (e.g., Qwen3.5-0.8B). Problem Statement The goal of this work is to accurately predict the end-to-end training time of a transformer-based LLM without executing the training process itself. Formally, given a model configuration (e.g., number of layers, hidden dimension, sequence length) and a target hardware platform (RTX 4090), we aim to estimate total training runtime for a specified number of tokens. This problem is challenging due to several factors: Heterogeneous operator mix: LLM training consists of diverse operators such as GEMMs, attention kernels, normalization, and activation functions, each with distinct performance characteristics. Nonlinear hardware behavior: GPU performance is influenced by memory hierarchy, kernel fusion, launch overheads, and scheduling effects that are difficult to capture analytically. Limitations of existing approaches: Simulation-based methods are slow and complex, while sampling-based methods require costly partial execution. We reformulate training-time prediction as a data-driven operator-level regression problem, where the runtime of each atomic operator is learned from micro-benchmark data and aggregated to estimate total training time. III. Proposed Approach Our approach models LLM training as a composition of atomic operators whose runtime can be predicted independently and then aggregated. The key insight is that transformer architectures exhibit a highly repetitive structure: each layer contains a fixed sequence of operations whose dimensions scale predictably with model configuration. This enables scalable prediction without enumerating the entire computation graph. To reduce complexity and improve practical usability, we focus on a single-GPU execution model and explicitly exclude distributed communication overheads. This design choice allows us to isolate core kernel-level behavior and build accurate runtime predictors using modest benchmarking effort. IV. Methodology We formalize the proposed framework as a six-stage pipeline. Step 1: Model Selection We begin by selecting a target LLM architecture, such as Qwen3.5–0.8B. The model configuration includes parameters such as number of layers, hidden dimension, sequence length, vocabulary size, and number of attention heads. Step 2: Operator Decomposition The selected model is decomposed into atomic operators, including: General matrix multiplications (GEMMs) Attention primitives (QKV projections, softmax) Normalization layers (LayerNorm or RMSNorm) Activation functions This decomposition converts the high-level model into a structured operator graph that mirrors real training execution. Operator Composition Reference from Empirical Training Runs To ground our operator-level abstraction in realistic workloads, we reference operator composition statistics extracted from a full pretraining run of a Qwen1.5–0.5B model executed on 8× NVIDIA A100-SXM4-80GB GPUs. In that run, embedding operations account for approximately 48.4% of total FLOPs, while attention and MLP blocks each contribute around 23.5%. Normalization and activation functions jointly account for less than 5% of total computation. This empirical distribution closely aligns with the operator structure assumed in our framework and justifies focusing modeling effort on GEMM-heavy and attention-related operators. Step 3: Micro-Benchmarking Each operator is independently benchmarked on the RTX 4090 across a wide range of configurations, varying: Batch size (b) Sequence length (l) Hidden dimension (d) Number of attention heads (h) The result is a dataset mapping operator configuration parameters to measured execution time. Benchmarking is performed in isolation to eliminate interference from unrelated kernels. Step 4: Regression Model Design We train machine-learning regression models to predict operator runtime from configuration parameters. Input features include both raw parameters (e.g., b, l, d, h) and derived quantities such as approximate FLOPs and memory footprint. Tree-based models such as Random Forest and XGBoost are used due to their ability to capture nonlinear interactions and piecewise behavior. Unlike analytical FLOP-based models, these regressors implicitly learn the impact of memory bandwidth limitations, kernel launch overheads, and hardware scheduling effects. Standard train–test splits are used to ensure generalization across unseen configurations. Step 5: Operator Runtime Aggregation Using the trained regressors, we predict runtime for each operator instance in the model. Since transformer layers repeat structurally, operator runtimes are multiplied by the number of layers to obtain per-batch execution time. Step 6: End-to-End Training Time Prediction Total training time is estimated by aggregating per-batch runtime across all batches and adding fixed system overheads such as GPU–CPU offloading: $T_{\text{total}} = L \times \sum T_{\text{op}} + T_{\text{overhead}}$ where $L$ is the number of layers and $T_{\text{op}}$ denotes operator runtime.
